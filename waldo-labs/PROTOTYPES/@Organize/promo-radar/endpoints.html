What Promo Radar needs from the Brand Intelligence API

Every endpoint Promo Radar reads — and where the gaps are

Promo Radar is built on the Brand Intelligence API. The keystone need — a structured promotion object on active ads — has now been added to /brands/{id}/ads/active, so discount detection no longer depends on regexing ad text. The remaining additions below are the roll-ups, history, and alert events that would let the dashboard stop fanning out across every brand and faking a promotion history.

Proposed additions

The keystone — a structured promotion object on ads — has landed on /brands/{id}/ads/active (shown first, for reference). The five additions after it build on that object: every roll-up, the history, the search, and the alerts derive from it.

FIELD · promotion object on ads ✓ Added · keystone

A classified promotion object is now attached to each ad on /brands/{id}/ads/active — so the prototype reads discount_type/discount_value/promo_code directly instead of regexing body, on_screen_text, and the landing offer string. Mirroring the same object onto /discover/ads would extend structured detection to cross-platform ad-library search.

{
  "ad_id": "ad_b8c4d1e2",
  "body": "The mid-season SALE is here. Up to 50% off...",
  "promotion": {
    "is_promotional": true,
    "discount_type": "percentage_off",   // percentage_off | dollar_off | bogo | free_shipping | free_gift | bundle | clearance | sitewide_sale
    "discount_value": 50,
    "discount_unit": "pct",            // pct | usd | other
    "discount_text": "Up to 50% off",
    "promo_code": null,
    "conditions": "While stocks last",
    "expires_at": "2026-05-31",
    "confidence": 0.94
  }
}

Powers: every live-promo card, the leaderboard ranking, search result labels, and the inputs to all four roll-ups below.

GET /brands/{id}/promotions ✗ Gap

A brand's promotion history — current and past — as discrete records with start/end dates, depth, type, channels, and the source ad/LP. Today the brand-detail timeline is stitched from lp-changes offer swaps plus ad start dates; this returns it cleanly.

{
  "brand_id": "brand_zara",
  "promotions": [
    {
      "promotion_id": "promo_91af",
      "discount_type": "sitewide_sale",
      "discount_value": 50,
      "label": "Mid-season sale",
      "started_at": "2026-05-21",
      "ended_at": null,        // null = still live
      "channels": ["meta", "google"],
      "source_ad_ids": ["ad_b8c4d1e2"]
    }
  ],
  "meta": { "total": 7 }
}

Powers: the brand-detail discount-history timeline and the depth-over-time series.

GET /categories/{id}/promotions/summary ✗ Gap

A category discount roll-up in one call: share of brands discounting, average depth, discount-type mix, and a week-over-week trend series. Today the prototype fans out across every brand in a category and aggregates client-side — an N×calls problem the dashboard repeats for all eight categories.

{
  "category_id": "cat_fashion",
  "window": "7d",
  "pct_brands_discounting": 0.48,
  "avg_depth_pct": 33,
  "wow_delta_pp": 9,
  "type_mix": { "percentage_off": 0.30, "sitewide_sale": 0.26, "clearance": 0.16 },
  "trend": [{ "week": "2026-05-25", "pct": 0.48 }]
}

Powers: the "discount pressure by industry" bar chart and the category-detail stats + type mix.

GET /discover/promotions ✗ Gap

Cross-index search for who's discounting right now: filter by category, discount type, and minimum depth; sort by depth or recency. This is the query the entire app is about — today it's impossible without pulling every brand's active ads and filtering locally.

GET /discover/promotions?active=true&min_depth=20&sort=depth_desc&category=cat_fashion

{
  "data": [
    {
      "brand_id": "brand_everlane",

      "category": "Apparel & Fashion",
      "discount_type": "clearance",
      "discount_value": 60,
      "discount_text": "Up to 60% off",
      "started_at": "2026-05-15"
    }
  ],
  "meta": { "total": 142 }
}

Powers: the "deepest discounts live now" leaderboard and category "discounting now" rosters.

GET /promotions/landscape ✗ Gap

A single top-level read for the home hero: a market-wide discount-pressure index, its week-over-week move, the deepest live discount, and a trend series — across the whole tracked universe, not one category.

{
  "as_of": "2026-05-28",
  "pct_brands_discounting": 0.36,
  "avg_depth_pct": 29,
  "wow_delta_pp": 8,
  "deepest_live": { "brand_name": "Wayfair", "discount_value": 70 },
  "trend": [{ "week": "2026-05-25", "pct": 0.38 }]
}

Powers: the four hero KPI cards and the 12-week discount-trend chart.

EVENTS · discount-scoped alert types ✗ Gap

New event_type values for POST /alerts so a user can track discounts directly, rather than leaning on the indirect competitor.messaging_pivot. With these, the alert form's three triggers map 1:1 onto API events, and /alerts/{id}/preview already gives the dry-run.

{
  "trigger": {
    "event_type": "promotion.deepened",   // promotion.started | promotion.ended | promotion.deepened | category.promo_surge
    "threshold": { "min_depth_pct": 30 }
  },
  "scope": { "brand_ids": ["brand_nike"] },
  "channels": [{ "type": "slack", "target": "#promo-watch" }]
}

Powers: the alert-setup triggers ("any discount live", "new discount starts", "deeper than X%") and the dry-run preview.
