What Prospector needs from the Brand Intelligence API

Every endpoint Prospector reads — and where the gaps are

Prospector turns an agency's past client roster into a weekly feed of look-alike brands that have an opening right now. Most of the raw material already exists in the Brand Intelligence API — brand & audience profiles, company contacts, hiring signals, category news, alerts, and pitch-pack generation. What's missing is the connective tissue: a way to ingest an agency profile, a multi-seed look-alike ranking that blends a whole client list, and a classified "opening" object so "why now" stops being stitched together client-side. The CRM-sync and LinkedIn-mutuals pieces sit outside the read API as connectors.

Proposed additions

The keystone is a POST /discover/lookalike-brands that blends a whole client roster into one ranked, scored list (shown first). Everything else either feeds it (agency profile, the openings object) or distributes it (events, the weekly digest). Two pieces are connectors, not read endpoints.

POST /discover/lookalike-brands ★ Keystone

Given a set of seed brand_ids (the agency's client roster) plus exclusions, return a single ranked, scored list of look-alike brands with structured match_reasons — which seed each one echoes and on which audiences. Today /audiences/{id}/brands does one audience and returns no score, so blending five clients + de-duping + ranking is all client-side.

POST /discover/lookalike-brands
{
  "seed_brand_ids": ["brand_olipop", "brand_allbirds", "brand_chime", "brand_sweetgreen", "brand_liquiddeath"],
  "exclude_brand_ids": [// current clients + conflicts],
  "limit": 5, "min_score": 70
}

{
  "data": [
    {
      "brand_id": "brand_poppi",
      "brand_name": "Poppi",
      "category": "Better-for-you beverage",
      "match_score": 94,
      "match_reasons": [
        { "echoes_seed": "brand_olipop", "shared_audience": "Gen Z wellness", "overlap": 0.92 }
      ]
    }
  ],
  "meta": { "candidates_scored": 3120 }
}

Powers: the weekly five, the match score on every card, and the "why related / echoes your client" block.

POST /agencies ✗ Gap

Ingest an agency at onboarding: a website to crawl plus an uploaded client list, returning the resolved seed brands and a derived audience + category fingerprint. There's no first-class "agency" object in the API today, so the whole setup flow has nowhere to persist.

POST /agencies
{
  "name": "Highline Studio",
  "website": "highlinestudio.com",
  "client_list": ["OLIPOP", "Allbirds", "Chime", "Sweetgreen", "Liquid Death"]
}

{
  "agency_id": "agency_highline",
  "resolved_clients": [{ "input": "OLIPOP", "brand_id": "brand_olipop" }],
  "fingerprint": {
    "audiences": ["Gen Z wellness", "Culture-forward", "Value-conscious"],
    "categories": ["Better-for-you CPG", "DTC apparel", "Consumer fintech"]
  }
}

Powers: the Setup screen, the sidebar agency identity, and the seed set passed to the look-alike matcher.

GET /brands/{id}/openings ✗ Gap

A classified array of "why now" openings per brand, each typed and sourced. Today this is reverse-engineered from /people diffs, /jobs, /news, and /spend-signal — this returns it pre-classified.

{
  "brand_id": "brand_vuori",
  "openings": [
    {
      "type": "leadership_change",    // funding | acquisition | agency_review | category_surge | campaign_lull
      "role": "cmo",
      "headline": "New CMO from Nike just started",
      "since": "2026-05-16",
      "source": { "type": "linkedin", "url": "…" },
      "confidence": 0.88
    }
  ]
}

Powers: the "Why now" box, every card's opening badge, and the prospect score's recency boost.

EVENTS · prospect-scoped alert types ✗ Gap

New event_type values for POST /alerts / GET /events so a tracked brand fires the moment an opening appears — instead of leaning on generic mention alerts. /alerts/{id}/preview already gives the dry-run.

{
  "trigger": {
    "event_type": "brand.leadership_change",  // brand.funding | brand.agency_review | prospect.new_opening
    "filter": { "role": "cmo" }
  },
  "scope": { "brand_ids": ["brand_immi", "brand_parade"] },
  "channels": [{ "type": "in_app" }, { "type": "email" }]
}

Powers: the Watchlist "opening signals" feed and the tracked-brand alerts.

POST /digests/prospect-brief ✗ Gap

The scheduled engine behind "This Week": every delivery_day, run the look-alike matcher for an agency, attach the freshest opening to each, and emit a batch of count prospects. Mirrors the existing /digests/monday-brief and /digests/audience-pulse shapes.

POST /digests/prospect-brief
{
  "agency_id": "agency_highline",
  "count": 5,
  "delivery_day": "monday",
  "require_opening": true     // only surface brands with a live opening
}

Powers: the weekly cadence, the "come back Monday" empty state, and delivery-cadence settings.

CONNECTOR · LinkedIn mutuals & HubSpot sync ✗ Out of read-API scope

Two features live outside the read API. LinkedIn mutuals needs each user's own OAuth relationship graph to answer "who do we already know here" — /companies/{id}/people gives the brand-side contacts, but the mutual edge is the customer's data, not ours. HubSpot sync is an outbound write/webhook that pushes accepted prospects into a customer's CRM as companies + contacts. Both are connector surfaces (OAuth + sync jobs), not endpoints we'd add to the bxcAPI itself — flagged here so the boundary is explicit.

Powers: the "N mutual connections" line on contacts, the warm-intro paths, and the Pipeline → HubSpot sync indicators.
