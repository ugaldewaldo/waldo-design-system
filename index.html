A component library and token system for Waldo's product surfaces. All components live in waldo-ui/src/components/ui/ and are exported from @waldo/ui.

Source of truth: waldo.tokens.json (Tokens Studio format) drives CSS, HTML, and Figma.

Component Rules
Conventions that apply to every component in the library.
Golden rule — atomic components are immutable

Atomic components (waldo-ui/src/components/ui/*.tsx) are frozen unless explicitly approved for change. The demo adapts to the component — never the reverse.

✅ Fix the demo HTML to match the component
❌ Change the component to match the demo
Zero hardcoded styles

Never write hex, rgba, or raw CSS variables in component code or demo HTML. Always use Tailwind classes that map to DS tokens.

❌ style="color:#32a9a9"
❌ className="text-[#32a9a9]"
✅ className="text-primary"

Figma
Figma-specific gotchas. The full sync flow lives in the Sync Runbook.
Sync flow → see Sync Runbook

The full token & component sync flow — code-first, with Figma variables written from code via the use_figma MCP (Tokens Studio is not used, variables are never edited by hand) — lives in the Sync Runbook docs page. This page covers only the Figma-specific gotchas below.

Figma model gotchas

Font weights are named strings: "Semi Bold" with space — not "600" or "SemiBold".

References do not resolve in composite textStyles — use literal values (Inter, "Regular", "14", "-2%"), not token references.

Top-level token-set keys map to Figma collections — don't add new ones without planning the full collection structure.

Sync Runbook
What to update everywhere when the design system changes — so nothing drifts out of sync.
Golden rule

waldo.tokens.json (tokens) and the .tsx components are the single source of truth. Everything else is derived. Direction is one-way: code → everything else. Figma variables are written from code via the use_figma MCP and are never edited by hand.

waldo.tokens.json  →  globals.css / waldo-shadcn-theme.css  →  .tsx  →  index.html (vanilla)  →  docs/token-catalog.yaml  →  Figma (via MCP)
Token change — new color or a value that changes
StepAction
1Edit waldo.tokens.json (source of truth)
2Regenerate globals.css + waldo-shadcn-theme.css → review affected .tsx
3Reflect in index.html vanilla (prototypes copy from here)
4Update docs/token-catalog.yaml — if it resolves something, move it out of known_discrepancies
5Write the variable to Figma via use_figma MCP — never by hand
6Publish the Figma library
7Consumer (e.g. Brand API prototypes) accepts library updates
8Review the docs page (see below)
9Validate: /ds-verify + AA contrast
Component change — new or modified
StepAction
1Create/edit the .tsx + its canonical class in index.html
2Entry in docs/usage-doctrine.yaml (no duplicates)
3Build it in the Figma Master DS, bound to variables (never hardcoded hex)
4Publish the Figma library
5Consumer accepts the update or drags in the new instance — never a local/pirate component
6Review the docs page (see below)
7Validate: /ds-verify
Update this docs page

The docs page is an artifact too — check it on every change.

The Doctrine and Token Catalog tabs render live from the YAML via fetch() — update the YAML and they refresh on reload, no manual edit.

Everything else (Overview, component demos, canonical classes like .btn-default) is manual — review and update by hand.

Anti-drift rules — the steps people forget
#Rule
1Publish the Figma library after every change — the #1 forgotten step. Without it, consumers see nothing.
2Accept updates in the consumer after publishing.
3Never edit a variable by hand in Figma → always waldo.tokens.json → MCP rewrites it. Hand edits get overwritten on the next sync.
4Keep index.html vanilla in step with the .tsx — prototypes copy from it.
5Move token-catalog.yaml discrepancies to resolved the moment a decision is made.
Enforcement & known gaps

The intelligence-layer tooling catches most violations, but a few checks are manual or not yet wired — know them so they don't bite:

MechanismWhat it catches / gap
tools/detect.jsReads token-catalog.yaml, so new catalog entries are caught automatically — except its hardcoded allow-lists (ALLOWED_HEX, rgba families, ALLOWED_RADII, ALLOWED_FONTS): a new family/value must be added to the JS by hand or it throws false positives.
tools/lint-doctrine.jsErrors if a .tsx has no usage-doctrine.yaml entry — but only if someone runs it (not yet wired to a hook). Infra-only files go in its EXEMPT set.
forbidden.patternsGap: documenting an anti-pattern in the YAML does not enforce it — detect.js regex rules are separate. Documented ≠ enforced.
Catalog derivationManual. Catalog says "derived from tokens.json" but nothing auto-syncs — change tokens.json and forget the catalog = silent drift.
YAML integrityA broken indent kills the live-rendered Doctrine / Token Catalog pages. Parse both YAML before commit.

Planned: a single pre-commit guard running detect.js + lint-doctrine.js + a YAML parse of both files — owned by this Sync Runbook.

Working rules (AI agents)

Verify visually before reporting. Never ask for review of a visual result without checking it with preview tools first. If the preview shows an error, the work is not done.

Commit before destructive ops. Before any git checkout or git reset --hard, commit everything first.

External comms need approval. Never post to Linear, Slack, or any external tool without explicit approval from the DS owner. Draft → wait → publish.

Language. All DS content (code, docs, Linear) in English, always.

[ ] Prop names don't clash with DOM attributes
[ ] Sub-component names match shadcn (DialogHeader not DialogHead)

Validators
Automated checks that catch DS violations — and the anti-pattern rules they enforce.
Edit hook — verify-component.sh

.claude/hooks/verify-component.sh runs automatically after every edit to waldo-ui/src/components/ files and index.html. Checks: hardcoded hex colors, wrong fonts, orphan styles, raw HTML in demos.

/ds-verify — manual audit before commit
1. Hardcoded colors — zero tolerance     grep hex, rgba(, bg-[#…]
2. Wrong fonts — Inter only              grep font-family
3. Legacy vars — must not exist          grep --txt-*, --fill-*, --line-*, --bg
4. Inline style= for colors/spacing      layout-only props allowed on demo wrappers
5. shadcn naming compliance              variants, props, sub-components, exports
Standalone detector — detect.js

A runnable script (pure Node, no dependencies, no LLM) that scans any file or folder for Waldo violations. Rules come from docs/token-catalog.yaml — add a rule to the catalog and the detector enforces it automatically.

node tools/detect.js <file|dir> [--json]

# examples
node tools/detect.js path/to/prototype/        # human-readable report
node tools/detect.js src/components/ --json    # JSON for CI
# exit code: 0 = clean · 1 = errors found
RuleCatches
model-defaultColors AI models generate when unconstrained (#09090b, #fafafa, #f5c453…) — each with its Waldo replacement
unknown-colorAny hex not in the token catalog, with the nearest Waldo token suggested
legacy-varVariables that don't exist in the DS (--txt-high, --bg, --fill-low…) with corrections
foreign-paletteTailwind default palette classes (text-red-500, bg-blue-600…)
wrong-font · italicNon-Inter fonts and any italic usage
raw-rgba · off-scale-radius · arbitrary-radiusAlphas outside the alpha namespace, radii off the token scale, rounded-[Npx]

Field-tested against the 4 waldo-labs prototypes (340 violations found) and against DS components (button.tsx: clean).

Anti-patterns — Colors
❌ Don't use green-500 as button fill with white text — contrast fails
❌ Don't use #ff6467 — that's shadcn's default red, not Waldo's coral (#de3a28)
❌ Don't use brand-* tokens in product UI — marketing only
❌ Don't use --destructive inside dropdown items — only in dialogs, alerts, destructive buttons
❌ Don't add red anywhere in product
Anti-patterns — Typography
❌ Don't use display-* typography inside product — UI track only
❌ Don't add italic anywhere
❌ Don't use monospace for numbers, values, labels in components
❌ Missing tracking-[-0.02em] on any UI text element
Anti-patterns — Components
❌ Don't use square checkboxes — always rounded-full
❌ Don't add font-semibold to buttons
❌ Don't add box-shadow ring to inputs
❌ Don't reference primitives directly from components — always via semantic tokens
❌ Don't add top-level keys to waldo.tokens.json without planning the Figma collection
❌ Don't write Figma textStyles with token references — resolve to literals first (Tokens Studio is out; Figma vars are written via use_figma MCP, never by hand)
Anti-patterns — SVG
❌ <svg stroke="var(--primary)">  — CSS vars don't resolve in SVG attributes

✅ <svg style="stroke:var(--primary)">
✅ <svg stroke="currentColor"> + parent color:var(--primary)
